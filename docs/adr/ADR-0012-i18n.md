# ADR-0012: Internationalization (de/en)

- **Status:** Approved
- **Date:** 2026-07-02
- **Decision IDs:** ADR-0012

## Context

TALOS targets German-speaking security/compliance users (the BSI library is German) as well as
English. As a static React app it needs a lightweight translation mechanism: simple string
catalogs, a `t(key)` lookup with an English fallback, and a per-user language preference — with
no server, middleware, or heavy i18n dependency.

## Decision

- **Languages:** `de`, `en` (extensible via a `SUPPORTED_LANGUAGES` list).
- **String catalogs:** one file per language, `src/locales/<lang>.json`
  (flat key → string) — human-readable, tooling-friendly, and native to a JS bundle.
- **Runtime:** a lightweight React **i18n context/provider** exposes `t(key, params?)`. No
  heavy dependency; a tiny helper with interpolation is sufficient. Missing key → English →
  raw key, so a missing translation never breaks the UI.
- **Preference:** stored in the `settings` object store (IndexedDB, ADR-0004), default `de`
  (BSI library content and the primary user base are German; matches `conventions.md` and
  `src/config.ts`'s `DEFAULT_LANGUAGE`); a settings dropdown changes it live. `<html lang>` is
  set from the active language.
- **Testability:** the `t()` context has a default value (used when no `<I18nProvider>` is
  mounted) that resolves against the **English** catalog — the same tier the missing-key
  fallback already uses. This lets isolated component tests render without wrapping every test
  in a provider, while the real app always mounts one provider at the root with the `de` default
  above.
- **Usage:** components call `t('compdef_title')`; no hard-coded user-facing strings.

## Consequences

**Positive:** adding a language = one `<lang>.json` + a list entry; EN fallback is safe;
consistent `t()` call sites are greppable.
**Negative:** every user-facing string must be keyed; catalogs must stay in sync (a lint/test
checks for missing keys across languages).
